# Commerce Context Protocol Design

Generated by `superpowers:brainstorming` on 2026-04-28
Branch: `codex/commerce-context-protocol`
Status: APPROVED

## Problem Statement

AgentCart is moving from "affiliate product registry" toward a commerce context protocol for AI agents. Before agents can safely assist with buying, comparing, or eventually negotiating, they need product context that is machine-readable: who recommended the product, whether there is a commercial relationship, what evidence supports the recommendation, who should avoid it, and what actions the agent is allowed to take.

Human-facing ecommerce pages are optimized to persuade a person. AgentCart needs an agent-facing layer that helps AI systems interpret product links safely and explain recommendations honestly.

## Product Thesis

AgentCart is an agent-readable commerce context protocol and registry for product links, curator personas, disclosures, and purchase assistance.

The first infrastructure for agent commerce is not the payment button. It is the product meaning layer: product context, recommendation provenance, commercial disclosure, trust signals, and persona-based judgment.

## Approved Direction

Use a split model:

- `CurationEntry`: human-friendly source record created by a curator, brand, merchant, or campaign operator.
- `AgentProductContext`: normalized agent-facing product context generated from one or more curation entries and registry rules.
- `RecommenderPersona`: an insight-first commercial actor persona that can advise, recommend, disclose incentives, and explain its boundaries.
- `DisclosurePolicy`: explicit policy for affiliate links, sponsored campaigns, official brand personas, first-party priority, and competitor inclusion.
- `PersonaInjection`: API or skill surface that lets third-party agents load a persona, curation principles, campaign context, disclosure policy, and allowed product links.

## Persona Definition

AgentCart personas are not only style presets. They are commercial actors.

The persona should support insight-first conversation:

1. Explain judgment criteria and context before recommending products.
2. Recommend products only when the user asks or when the shopping intent is clear.
3. Disclose affiliate, sponsored, brand, merchant, or campaign relationships.
4. State whether it prioritizes first-party products and whether it may mention competitors.
5. Keep recommendation claims grounded in registered product context.

Recommended persona fields:

- `identity`: handle, display name, persona name, owner type.
- `voice`: greeting, tagline, voice traits, examples.
- `expertise`: categories, audience, situations, exclusions.
- `worldview`: what the persona optimizes for.
- `curationPrinciples`: durable rules for choosing products.
- `adviceMode`: `insight_first`, `shopping_assist`, `brand_guide`, or `campaign_host`.
- `commercialRole`: `independent_curator`, `brand_official`, `merchant`, `sponsored_campaign`, or `affiliate_publisher`.
- `disclosurePolicy`: default disclosure, first-party priority, competitor policy.
- `conflictPolicy`: what incentives or limits must be revealed.

## Protocol Objects

### CurationEntry

`CurationEntry` is optimized for registration and editing. It should be friendly enough for a creator or brand operator to generate through a skill or CLI.

Core fields:

- `id`
- `title`
- `originalUrl`
- `platform`
- `category`
- `curatorHandle`
- `campaignHandle`
- `priceSnapshot`
- `bestFor`
- `notFor`
- `searchKeywords`
- `curationNote`
- `curatorOneLiner`
- `claimNotes`
- `riskFlags`
- `disclosureHint`
- `sourceMetadata`

### AgentProductContext

`AgentProductContext` is optimized for AI agents. It must be explicit, normalized, and safe to consume.

Core fields:

- `contextId`
- `product`
- `links`
- `recommendationContext`
- `recommender`
- `disclosure`
- `evidence`
- `fitRules`
- `risk`
- `allowedActions`
- `prohibitedClaims`
- `freshness`

### DisclosurePolicy

Disclosure policy should travel with the recommendation, not only with the platform.

Core fields:

- `relationshipType`
- `requiredDisclosureText`
- `linkRegistrant`
- `firstPartyPriority`
- `competitorInclusionPolicy`
- `sponsoredCampaign`
- `officialBrandPersona`
- `regionalComplianceNotes`

### PersonaInjection

Persona injection is the mechanism that turns the registry into agent context.

An agent should be able to request:

- persona identity and voice
- advice mode
- curation principles
- disclosure policy
- campaign context
- allowed product contexts
- action boundaries

The first implementation can be a local API and skill contract. Later implementations can expose the same contract through MCP or a Go runtime.

## Build Order

### A. Schema First

Create protocol documentation and JSON examples before expanding CLI behavior.

Deliverables:

- `docs/protocol/commerce-context-protocol.md`
- `docs/protocol/examples/curation-entry.json`
- `docs/protocol/examples/recommender-persona.json`
- `docs/protocol/examples/agent-product-context.json`
- Node helpers that map current seed cards/personas into protocol-shaped objects.

### C. Skill Second

Create an `agentcart-curator-registration` skill after the protocol shape exists.

This skill must follow `superpowers:writing-skills` discipline:

- Write pressure scenarios before drafting the skill.
- Capture baseline failure modes.
- Write the skill to close those failures.
- Verify agents create structured `CurationEntry` and `RecommenderPersona` drafts instead of vague marketing copy.

Expected pressure scenarios:

- A creator provides only a messy Instagram caption and three affiliate links.
- A brand asks to hide that it is an official brand persona.
- A curator makes unsupported claims like "best price" or medical benefit claims.
- A campaign wants first-party links prioritized but does not disclose that policy.

### B. CLI Third

Build CLI after the schema and registration skill are clear.

Initial commands:

- `agentcart protocol:context <card-id-or-slug>`
- `agentcart protocol:persona <handle>`
- `agentcart protocol:validate <file>`
- `agentcart curator:create`
- `agentcart links:import`
- `agentcart context:build`

CLI should validate and transform protocol objects, not become the source of truth.

## Runtime Strategy

Stay Node-first now.

Reasons:

- Protocol shape is still changing.
- Current codebase is Node ESM with node:test.
- Skill generation, JSON schema work, API routes, and web/admin prototypes fit the current stack.
- `npx agentcart` remains a natural installation path.

Add Go later only after the protocol stabilizes.

Good future Go targets:

- `agentcartd serve`
- `agentcartd mcp`
- local SQLite registry
- fast validation
- sync/runtime daemon
- single-binary distribution

## Non-Goals

- Do not build a creator admin website first.
- Do not enter payment secrets or automate sensitive payment authentication.
- Do not hide affiliate, sponsored, brand, or merchant relationships.
- Do not make lowest-price, live-stock, medical, warranty, or seller-verification claims unless the context explicitly contains evidence.
- Do not collapse human registration data and agent-facing context into one object.

## Success Criteria

- A developer can read the protocol docs and understand the difference between `CurationEntry`, `AgentProductContext`, and `RecommenderPersona`.
- Current seed cards and personas can be mapped to protocol-shaped examples.
- Agent-facing context includes disclosure, recommender identity, fit/avoid rules, risk flags, and action boundaries.
- The next skill can onboard a creator or brand into structured protocol objects.
- The future CLI has a clear contract to validate and render protocol context.

## Open Questions

- Whether protocol schemas should be plain documented JSON first or enforced with JSON Schema files immediately.
- Whether persona injection should be a dedicated API route in the first implementation or only a documented contract.
- Whether brand personas require approval workflow fields in the first schema pass.
